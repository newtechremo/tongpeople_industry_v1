> **서비스명:** 산업현장통 2.0 - 통패스 (TongPass)

| 버전 | 작성일 | 작성자 | 비고 |
|------|--------|--------|------|
| **v1.0** | 2026.01.16 | PM | 초안 작성 (쏘카 비즈니스 벤치마킹) |

---

## 1. 개요 (Overview)

### 1.1 목적

기존 "한 명 초대하기(선등록)" 방식을 유지하면서, **회사 코드/QR 기반 직접 가입** 기능을 추가하여 대규모 인원의 효율적인 가입을 지원한다.

### 1.2 기대 효과

| 항목 | AS-IS (선등록 방식) | TO-BE (코드/QR 방식 추가) |
|------|---------------------|---------------------------|
| 등록 주체 | 관리자가 1명씩 등록 | 근로자가 직접 가입 |
| 관리자 업무량 | 인원수 x 입력 시간 | 코드 공유 1회 + 승인만 |
| 가입 속도 | 느림 (순차 등록) | 빠름 (동시 가입 가능) |
| 적합 시나리오 | 소수 핵심 인력 | 대규모 현장 인원 |

### 1.3 가입 방식 비교표

| 구분 | 방식 A: 선등록 | 방식 B: 회사코드/QR 가입 |
|------|---------------|------------------------|
| **등록 주체** | 관리자 | 근로자 본인 |
| **진입 경로** | 관리자가 정보 입력 -> 문자 발송 | 근로자가 코드 입력 또는 QR 스캔 |
| **정보 입력** | 관리자가 입력 | 근로자가 직접 입력 |
| **초기 상태** | `PENDING` (동의대기) | `REQUESTED` (승인대기) |
| **최종 활성화** | 근로자 동의 시 즉시 `ACTIVE` | 관리자 승인 후 `ACTIVE` |
| **권한 설정** | 관리자가 TEAM_ADMIN 가능 | WORKER 고정 |
| **적합 대상** | 핵심 인력, 팀 관리자 | 일반 근로자 대규모 가입 |

## 2. 회사 코드 체계

### 2.1 코드 구조

```
회사코드 (6자리 영숫자)
- 형식: [A-Z0-9]{6}
- 예시: A1B2C3, O6N511, XY7890
- 유효기간: 무제한 (활성/비활성 관리)
- 고유성: 전체 시스템 내 UNIQUE
```

> **기존 문서와의 정합성:**
> - `docs/figma/screen-specs/01-company-code.md`에서 4~20자 가변으로 정의되어 있음
> - **본 기획서 기준 6자리 고정으로 통일** (쏘카 벤치마킹, 사용자 입력 편의성)
> - Figma 스펙 문서 업데이트 필요: `minLength: 6, maxLength: 6`

### 2.2 코드 생성 규칙

| 항목 | 규칙 | 설명 |
|------|------|------|
| 길이 | **6자리 고정** | 입력 편의성, 쏘카 벤치마킹 |
| 문자 구성 | 대문자 + 숫자 | 가독성, 혼동 방지 (O/0, I/1 허용) |
| 생성 방식 | 자동 생성 | 랜덤 생성 후 중복 체크 |
| 재생성 | 가능 | 기존 코드 비활성화 후 신규 발급 |
| 유효기간 | 무제한 | 관리자가 수동 비활성화 |

### 2.3 코드-현장 연결 구조

회사 코드는 **회사(Company) 단위**로 발급하고, 현장이 여러 개면 근로자가 앱에서 선택한다.

- 현장이 1개인 경우: 자동으로 해당 현장 선택
- 현장이 2개 이상인 경우: 근로자가 현장 목록에서 선택

## 3. 기능 명세서 (관리자 웹)

### 3.1 근로자 관리 페이지 개편

#### 3.1.1 페이지 구조 (버튼 추가 방식)

**[근로자 관리] 페이지 - 현재 구조 유지 + 버튼 1개 추가**

```
┌─────────────────────────────────────────────────────────────────┐
│  근로자 관리                                                     │
│  총 12명의 근로자                                                │
│                                                                  │
│  [QR/코드 공유]  [엑셀 일괄 등록]  [신규 동의링크]               │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ 🔍 이름, 연락처 검색  │ 전체 팀 ▼ │ 전체 역할 ▼ │ 전체 상태 ▼ │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  [테이블: 기존 마스터 테이블 유지]                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

> **참고:** 탭 네비게이션 대신 기존 **상태 필터 드롭다운**을 활용합니다.
> - 상태 필터에 `REQUESTED`(승인대기) 옵션 추가
> - 승인대기 근로자가 있으면 필터 옆에 빨간 뱃지 표시

#### 3.1.2 상단 버튼 변경

| AS-IS | TO-BE |
|-------|-------|
| - | **[QR/코드 공유] (신규, 좌측 추가)** |
| [엑셀 일괄 등록] | [엑셀 일괄 등록] (유지) |
| [신규 동의링크] | [신규 동의링크] (유지) |

#### 3.1.3 상태 필터 옵션 확장

```typescript
// AS-IS
type StatusFilter = 'ALL' | 'PENDING' | 'INACTIVE';

// TO-BE
type StatusFilter = 'ALL' | 'PENDING' | 'REQUESTED' | 'INACTIVE' | 'BLOCKED';
```

| 옵션 | 라벨 | 설명 |
|------|------|------|
| `ALL` | 전체 상태 | 기본값 |
| `PENDING` | 동의대기 | 관리자 선등록 후 동의 전 |
| **`REQUESTED`** | **승인대기** | **근로자 직접가입, 관리자 승인 전 (신규)** |
| `INACTIVE` | 비활성 | 퇴사/비활성 처리 |
| `BLOCKED` | 차단 | 관리자 차단 |

## 5. 상태 흐름도

### 5.1 전체 가입 플로우

```
                     가입 방식 선택
         |                   |                    |
         v                   v                    v
   방식 A              방식 B               방식 C
   관리자 선등록       코드/QR 가입         엑셀 일괄 등록
         |                   |                    |
         v                   v                    v
   관리자가 정보       근로자가 코드        관리자가 엑셀
   입력 + SMS 발송     입력 or QR 스캔      업로드
         |                   |                    |
         v                   v                    v
   PENDING             근로자가 정보        PENDING
   (동의대기)          입력 + 서명          (동의대기)
         |                   |                    |
         v                   v                    |
   근로자가 링크       REQUESTED                  |
   클릭 + 동의         (승인대기)                 |
         |                   |                    |
         |                   v                    |
         |             관리자가 승인               |
         |             (팀 배정)                   |
         |                   |                    |
         v                   v                    v
                     ACTIVE (정상)
                     서비스 이용 가능
```

### 5.2 상태값 정의

```typescript
type WorkerStatus =
  | 'PENDING'    // 동의대기 - 관리자 선등록, 근로자 동의 전
  | 'REQUESTED'  // 승인대기 - 근로자 직접가입, 관리자 승인 전
  | 'ACTIVE'     // 정상 - 서비스 이용 가능
  | 'REJECTED'   // 반려 - 가입 거절 (재가입 가능)
  | 'INACTIVE'   // 비활성 - 퇴사 처리
  | 'BLOCKED';   // 차단 - 재가입 불가
```

### 5.3 상태 전이 규칙

| 현재 상태 | 이벤트 | 다음 상태 | 주체 |
|-----------|--------|-----------|------|
| (신규) | 관리자 선등록 | PENDING | 관리자 |
| (신규) | 근로자 직접가입 | REQUESTED | 근로자 |
| PENDING | 근로자 동의 완료 | ACTIVE | 근로자 |
| REQUESTED | 관리자 승인 | ACTIVE | 관리자 |
| REQUESTED | 관리자 반려 | REJECTED | 관리자 |
| REQUESTED | 관리자 차단 | BLOCKED | 관리자 |
| ACTIVE | 관리자 비활성화 | INACTIVE | 관리자 |
| ACTIVE | 관리자 차단 | BLOCKED | 관리자 |
| INACTIVE | 관리자 재활성화 | ACTIVE | 관리자 |
| REJECTED | 근로자 재가입 | REQUESTED | 근로자 |

## 6. API 명세

### 6.1 회사 코드 관리 API

#### 6.1.1 회사 코드 조회
```
GET /api/admin/company-code
Authorization: Bearer {accessToken}
```

**Response:**
```json
{
  "success": true,
  "data": {
    "code": "A1B2C3",
    "companyId": "uuid",
    "companyName": "(주)리모",
    "isActive": true,
    "createdAt": "2026-01-15T09:00:00Z",
    "qrImageUrl": "https://..."
  }
}
```

#### 6.1.2 회사 코드 재생성
```
POST /api/admin/company-code/regenerate
Authorization: Bearer {accessToken}
```

**Response:**
```json
{
  "success": true,
  "data": {
    "code": "X7Y8Z9",
    "createdAt": "2026-01-16T10:00:00Z"
  }
}
```

#### 6.1.3 회사 코드 활성/비활성
```
PATCH /api/admin/company-code/status
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request:**
```json
{
  "isActive": false
}
```

## 7. DB 스키마 변경사항

### 7.1 company_codes 테이블 (신규)

```sql
CREATE TABLE company_codes (
  id BIGSERIAL PRIMARY KEY,
  company_id BIGINT NOT NULL REFERENCES companies(id) ON DELETE CASCADE,
  code VARCHAR(6) NOT NULL UNIQUE,
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  deactivated_at TIMESTAMPTZ,

  CONSTRAINT valid_code_format CHECK (code ~ '^[A-Z0-9]{6}$')
);

CREATE INDEX idx_company_codes_company ON company_codes(company_id);
CREATE INDEX idx_company_codes_code ON company_codes(code) WHERE is_active = TRUE;
```

### 7.2 users 테이블 확장

```sql
-- 기존 users 테이블에 컬럼 추가
ALTER TABLE users ADD COLUMN status worker_status DEFAULT 'ACTIVE';
ALTER TABLE users ADD COLUMN requested_team_id BIGINT REFERENCES partners(id);
ALTER TABLE users ADD COLUMN rejection_reason TEXT;
ALTER TABLE users ADD COLUMN requested_at TIMESTAMPTZ;
ALTER TABLE users ADD COLUMN approved_at TIMESTAMPTZ;
ALTER TABLE users ADD COLUMN approved_by UUID REFERENCES users(id);

-- 상태 ENUM 타입 생성
CREATE TYPE worker_status AS ENUM (
  'PENDING',    -- 동의대기
  'REQUESTED',  -- 승인대기
  'ACTIVE',     -- 정상
  'REJECTED',   -- 반려
  'INACTIVE',   -- 비활성
  'BLOCKED'     -- 차단
);
```

## 1. 개요 (Overview)

| 항목 | 내용 |
|------|------|
| **프로젝트명** | 통패스 근로자 앱 (TongPass Worker App) |
| **플랫폼** | React Native (iOS / Android) |
| **목적** | 폐쇄형 건설 현장 근로자의 간편 가입 및 출퇴근 기록 |

### 핵심 원칙

1. **회사 코드 기반 진입:** 코드를 모르면 가입 불가
2. **두 가지 가입 방식 병행:**
   - **관리자 선등록 (방식 A):** 관리자가 먼저 등록 → 근로자 동의 → 즉시 ACTIVE (프리패스)
   - **근로자 직접가입 (방식 B):** 근로자가 직접 가입 → `REQUESTED` 상태 → 관리자 승인 후 ACTIVE
3. **간소화:** 비밀번호 없음(휴대폰 인증), 이메일 선택 입력

## 2. 조직 체계 (CLAUDE.md 기준)

조직의 계층을 **회사 > 현장 > 팀(업체)** 3단계로 고정합니다.

```
회사 (Company)
└── 현장 (Site)           예: 대전공장, 서울본사
    └── 팀 (Team/Partner)  예: (주)정이앤지, 협력업체A
        └── 근로자 (Worker)
```

### 사용자 역할 (4단계)

| 역할 | 명칭 | 범위 | 주요 권한 |
|------|------|------|-----------|
| **최고 관리자** | 회사/본사 | 회사 전체 | 시스템 전체 설정, 결제 관리, 모든 현장 접근 |
| **현장 관리자** | 현장 소장 | 특정 현장 | 해당 현장의 모든 데이터 관리, 팀/근로자 관리 |
| **팀 관리자** | 업체장/오반장 | 특정 팀 | 자기 팀원들의 출퇴근 QR 스캔, 팀원 조회 |
| **근로자** | 팀원 | 본인 | QR 생성, 본인 출퇴근 인증 및 기록 조회 |

```typescript
type UserRole = 'SUPER_ADMIN' | 'SITE_ADMIN' | 'TEAM_ADMIN' | 'WORKER';
```

> **본 PRD 범위:** 근로자(WORKER) 역할의 앱 기능만 다룹니다.

### 근로자 상태값 (WorkerStatus)

```typescript
type WorkerStatus = 'PENDING' | 'REQUESTED' | 'ACTIVE' | 'INACTIVE' | 'BLOCKED';
```

| 상태코드 | 한글명 | 배지색 | 발생 경로 | 관리자 액션 |
|----------|--------|--------|----------|------------|
| `PENDING` | 동의대기 | 노랑 | 관리자 선등록 | SMS 재전송 |
| `REQUESTED` | 승인대기 | 빨강 | 근로자 직접가입 | [승인] / [반려] |
| `ACTIVE` | 정상 | 없음 | 가입완료 | 정보수정 / 비활성화 |
| `INACTIVE` | 비활성 | 회색 | 퇴사/차단 | 활성화 |
| `BLOCKED` | 차단 | 검정 | 관리자 차단 | - |

## 3. 퇴근 모드 및 근무일 사이클

### 퇴근 모드

| 모드 | 설명 |
|------|------|
| `AUTO_8H` | 출근 후 8시간 경과 시 자동 퇴근 처리 |
| `MANUAL` | 근로자가 직접 퇴근 버튼을 눌러야 퇴근 처리 |

> **📌 설정 권한:** 퇴근 모드는 **현장(Site) 단위 설정**이며, **관리자 웹**에서만 변경 가능합니다. 근로자 앱에서는 현장 설정값을 조회만 합니다.

### 근무일 사이클

- **시작:** 당일 04:00
- **종료:** 익일 03:59

> 예: 2026-01-13 04:00 ~ 2026-01-14 03:59가 하나의 "근무일"입니다.

## 4. 입력 항목 비교표 (핵심!)

### 방식 A: 관리자 선등록 (PC 웹)

관리자가 근로자를 미리 등록할 때 입력하는 항목입니다.

| 항목 | 타입 | 필수 | 기본값 | 설명 |
|------|------|:----:|--------|------|
| **소속 팀 (업체)** | Dropdown | O | - | 사전 등록된 팀 목록에서 선택 |
| **이름** | Text | O | - | 근로자 실명 |
| **휴대폰 번호** | Tel | O | - | 동의 링크 발송용 (고유 키) |
| **생년월일** | Number (8자리) | O | - | YYYYMMDD 형식, 만 65세 자동 판별용 |
| **직책/직종** | Text + Datalist | O | - | 선택 또는 직접 입력 가능 |
| **시스템 권한** | Radio | O | 일반 근로자 | 팀 관리자(TEAM_ADMIN) vs 일반 근로자(WORKER) |
| **국적** | Select | - | 대한민국 | 국가 선택 |
| **성별** | Toggle | - | 남성 | 남성/여성 |

#### 직책/직종 선택 옵션
- 공사기사, 전기기사, 미장기사, 설비기사, 안전관리자, 일반근로자, 기타

#### 국적 선택 옵션
- 대한민국, 중국, 베트남, 네팔, 미얀마, 캄보디아, 태국, 인도네시아, 우즈베키스탄, 기타

### 방식 B: 근로자 직접가입 (모바일 앱)

근로자가 앱에서 직접 입력하는 항목입니다.

| 단계 | 항목 | 타입 | 필수 | 설명 |
|------|------|------|:----:|------|
| 3단계 | **휴대폰번호** | Tel | O | SMS 인증 필수, ID 역할 |
| 5단계 | **이름** | Text | O | 실명 |
| 5단계 | **생년월일** | Number (8자리) | O | YYYYMMDD, 고령자(65세+) 판별용 |
| 5단계 | **이메일주소** | Email | X | **선택 (빈값 허용)**, 추후 소셜 로그인 연동용 |
| 5단계 | **성별** | Radio | O | 남성 / 여성 |
| 5단계 | **국적** | Dropdown | O | 국가 선택 |
| 5단계 | **소속 팀** | Dropdown | O | 현장의 팀 목록에서 선택 |
| 5단계 | **직책/직종** | Text + Datalist | O | 선택 또는 직접 입력 |
| 6단계 | **약관동의** | Checkbox | O | 4개 항목 필수 |
| 7단계 | **전자서명** | Canvas | O | Base64 이미지 저장 |

> **📌 시스템 권한:** 앱에서 직접 가입 시 **WORKER(일반 근로자) 고정**입니다. 팀 관리자 권한은 관리자가 PC에서만 부여할 수 있습니다.

### 방식 A vs B 비교 요약

| 항목 | 방식 A (선등록) | 방식 B (직접가입) | 비고 |
|------|:--------------:|:----------------:|------|
| 소속 팀 | O (관리자 지정) | O (앱에서 선택) | 통일됨 |
| 이름 | O | O | |
| 휴대폰번호 | O | O | 고유 키 |
| 생년월일 | O | O | 필수, 고령자 판별 |
| 직책/직종 | O | O (앱에서 입력) | 통일됨 |
| 시스템 권한 | O (선택 가능) | X (WORKER 고정) | 보안상 적절 |
| 이메일 | X | O (선택) | 추후 소셜 로그인용 |
| 성별 | O | O | |
| 국적 | O | O | |
| 약관동의 | O (링크) | O (앱) | |
| 전자서명 | O | O | |

> **핵심 차이점:** 시스템 권한만 다름 (방식 A는 TEAM_ADMIN 가능, 방식 B는 WORKER 고정)

## 5. 사용자 플로우 (User Flow)

```mermaid
graph TD
    Start[앱 실행] --> LoginCheck{로그인 상태?}

    LoginCheck -- Yes (상태: ACTIVE) --> Main[메인: 출퇴근]
    LoginCheck -- Yes (상태: REQUESTED) --> Wait[승인 대기 화면]
    LoginCheck -- No --> Step1[1. 회사코드 입력]

    Step1 --> Step2[2. 휴대폰 번호 인증]
    Step2 --> ExistingCheck{기존 회원인가?}

    ExistingCheck -- Yes --> Recover[계정 복구 및 자동 로그인]
    ExistingCheck -- No --> PreCheck{관리자 선등록 명단에 있는가?}

    PreCheck -- Yes (선등록) --> AutoFill[3. 정보 입력 (자동 완성)]
    PreCheck -- No (직접가입) --> ManualFill[3. 정보 입력 (직접 입력)]

    AutoFill -- 데이터 수정 발생 --> Conflict[데이터 불일치 플래그 ON]
    AutoFill & Conflict --> Step4A[4. 약관 동의 & 전자서명]
    ManualFill --> Step4B[4. 약관 동의 & 전자서명]

    Step4A --> DirectActive[즉시 ACTIVE - 프리패스]
    DirectActive --> Main

    Step4B --> RequestDone[가입 요청 완료]
    RequestDone --> Wait[5. 승인 대기 화면 - REQUESTED]

    Wait -- 관리자 승인 완료 --> Main
```

### 핵심 분기점: 선등록 여부

| 조건 | 상태 흐름 | 승인 필요 | 설명 |
|------|----------|:--------:|------|
| 관리자가 미리 등록한 번호 | `PENDING`(동의대기) → 동의 완료 → `ACTIVE` | X | 관리자가 이미 신원 확인함 (프리패스) |
| 근로자가 직접 가입한 번호 | `REQUESTED`(승인대기) → 관리자 승인 → `ACTIVE` | O | 누군지 모르므로 관리자 확인 필요 |

#### 선등록 데이터 매칭 (Pre-fill)

- API 응답에 `preRegisteredData`가 있다면, 입력 폼의 기본값으로 설정
- **안내 메시지:** "관리자가 등록한 정보를 불러왔습니다." (상단 배너)

#### 데이터 충돌 처리 (Conflict Logic)

1. **팝업:** "관리자 등록 정보와 다릅니다. 입력한 정보로 가입하시겠습니까?"
2. **[확인] 시:** 서버에 `is_data_conflict: true` 플래그 전송
3. **관리자 확인:** PC 관리자 페이지에서 "정보 불일치" 표시됨

### workers 테이블

| 컬럼명 | 타입 | 설명 |
|--------|------|------|

| `gender` | CHAR(1) | M/F |
| `nationality` | VARCHAR | 국적 코드 |

#### 고령자(is_senior) 판별 로직

```sql
-- PostgreSQL Trigger (Supabase)
CREATE OR REPLACE FUNCTION calculate_is_senior()
RETURNS TRIGGER AS $$
BEGIN
  NEW.is_senior := (
    EXTRACT(YEAR FROM AGE(CURRENT_DATE, TO_DATE(NEW.birth_date, 'YYYYMMDD'))) >= 65
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_calculate_is_senior
BEFORE INSERT OR UPDATE OF birth_date ON workers
FOR EACH ROW EXECUTE FUNCTION calculate_is_senior();
```

> **📌 참고:** 생일이 지나 65세가 되는 경우는 연 1회 수동 갱신 스크립트로 처리합니다. (MVP 단순성 우선)
